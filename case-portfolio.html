01 — contexto

Por que HTML puro
em 2026.

A decisão de não usar framework foi intencional — não por nostalgia ou limitação técnica, mas por eficiência real. Um portfólio estático que não precisa de build, não tem dependências que quebram e pode ser editado em qualquer editor de texto é mais durável do que um projeto React que entra em conflito de versões em 6 meses.

A outra razão: HTML/CSS/JS são a linguagem nativa do Claude Code. Sem camada de abstração entre a instrução e o resultado, o loop de feedback é imediato — escrevo o que quero, vejo na tela, itero.

O portfólio não é apenas o resultado do processo — é evidência do processo. Cada decisão de estrutura, cada componente, cada case page documenta uma habilidade real aplicada a um projeto real.

03 — o processo de construção

Como o Claude Code
foi usado.

O Claude Code não foi usado como gerador de código — foi usado como par de programação. A diferença é que par de programação implica diálogo: você traz o contexto, o par questiona, itera, sugere alternativas.

01

Definição da identidade visual

Paleta (purple/green/cream/charcoal), tipografia (Cooper Black + DM Mono + Cabin) e sistema de componentes foram definidos em sessão — não como template, mas como decisões justificadas a cada escolha.

02

Hero e navegação

O hero foi construído em grid 50/50 — conteúdo editorial à esquerda, seção verde com mascote à direita. Cursor customizado, animação de marquee, transição suave entre seções via showPage().

03

Cases de conteúdo

Cada case foi construído separadamente com o mesmo sistema de CSS inline — sem arquivo externo para manter os arquivos autossuficientes. Carrosséis com scroll-snap, drag-to-scroll e botões de navegação com JS mínimo.

04

Ferramentas com API

As ferramentas funcionais (análise de métricas, organizador, foco) usam a Claude API via Netlify Functions como proxy — a chave fica no servidor, não no HTML público. Solução encontrada iterativamente após a versão inicial expor a chave no código.

05

Correções e refinamentos contínuos

Grid de métricas que transbordava no mobile, slides de carrossel trocados, títulos incorretos, imagens com caminhos quebrados — cada bug foi identificado, documentado e corrigido em sessão. O histórico de commits é o log do processo.

04 — decisões técnicas

O porquê de cada
escolha.

CSS inline em vez de arquivo externo

Cada arquivo HTML é autossuficiente — abre em qualquer contexto sem depender de outros arquivos. Para um portfólio estático hospedado via drag-and-drop, isso elimina uma classe inteira de problemas de caminho de arquivo.

JS mínimo, sem bibliotecas

Os carrosséis, filtros de portfólio e navegação entre páginas foram implementados com JS vanilla. Sem jQuery, sem lodash, sem dependência que pode deixar de ser mantida. O código é legível, editável e não tem surpresas de versão.

Netlify Functions como proxy de API

Expor a chave da Claude API no HTML público é um risco de segurança real — qualquer um que inspecione o código pode usar a chave. O proxy via Netlify Functions mantém a chave no servidor e expõe apenas o endpoint, sem mudança de experiência para o usuário.

Git como sistema de deploy, não só versionamento

Cada mudança vira um commit com mensagem descritiva. O histórico do repositório não é só backup — é documentação do processo de construção. Qualquer pessoa pode ver o que mudou, quando e por quê.

06 — aprendizados

✦ o que esse projeto ensinou

Além do
portfólio.

  • IA como par de programação é diferente de IA como gerador. A diferença está em quem traz o contexto, as restrições e as decisões. Quando você sabe o que quer, a IA executa melhor. Quando você não sabe, a IA vai no sentido errado.
  • HTML puro envelhece melhor do que frameworks. A ausência de dependências é um ativo. Em dois anos, este portfólio vai abrir exatamente do mesmo jeito que hoje.
  • Documentar o processo é parte do trabalho, não extra. Os cases de ferramenta, os commits descritivos, os comentários de decisão — tudo isso é evidência de raciocínio que o resultado final não mostra.
  • Bugs são dados de design. O grid que transbordava no mobile, os slides trocados — cada bug revelou uma suposição incorreta sobre como o conteúdo seria acessado. Corrigi-los é parte do processo de design, não falha de execução.
